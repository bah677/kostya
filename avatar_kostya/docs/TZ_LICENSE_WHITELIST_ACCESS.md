# Техническое задание: переход с чёрного списка на белый список по таблице `license`

**Проект:** репозиторий `avatar` (Telegram-бот на aiogram 3).  
**Цель:** вместо модели «все могут пользоваться ботом, кроме забаненных» ввести модель «ботом могут пользоваться **только** пользователи с **действующей записью в таблице лицензий**», с сохранением узкого набора **публичных** маршрутов (онбординг, оплата, поддержка).  
**Фичи:** отключение отдельных фич достигается **неназначением** их в `BotApplication._register_features` (например `referral_program`, `support` — по необходимости).

---

## 1. Контекст и проблема

### 1.1. Как сейчас

| Компонент | Поведение |
|-----------|-----------|
| `bot/middleware/access_control.py` | После логирования входа вызывает политику `decide` → при `DENY` отвечает пользователю и **не** передаёт апдейт дальше. |
| `bot/base_app.py` | Подключается `AccessControlMiddleware(policy=BanBlacklistPolicy(self.user_storage))`. |
| `bot/access/policies.py` — `BanBlacklistPolicy` | Публичные команды (например `/start`, `/support`, `/payment`, …) проходят **без проверки**. Все остальные маршруты: если в `users` у пользователя `is_banned` — **DENY**, иначе **ALLOW**. |
| `bot/features/license_service.py` | `check_access` / `can_ask_question` фактически завязаны на **бан**, а не на строку в `license`. Для пользователя без строки в БД доступ **разрешён** («создастся при онбординге»). |
| `storage/db/licenses.py` | Есть полноценная работа с таблицей `license`: активная лицензия, `expires_at`, `get_user_active_license`, `list_user_ids_with_active_license`, и т.д. |
| `storage/db/users.py` — `check_user_license` | Проверяет только `status = 'active'`, **без** проверки срока `expires_at` (устаревшая/неполная семантика относительно `LicensesMixin`). |

Итог: таблица `license` используется для подписок и продлений, но **не** как единственный источник права на использование бота в middleware.

### 1.2. Желаемое поведение

- **Белый список:** доступ к **непубличным** маршрутам только если пользователь имеет **действующую лицензию** в смысле бизнес-логики уже реализованной в `LicensesMixin` (активный статус и не истёкший срок — см. п. 3).
- **Бан сохраняется:** пользователь с `is_banned = true` **никогда** не получает полный доступ, даже при наличии записи в `license` (бан строже лицензии). Публичные маршруты для забаненного — по решению п. 4.3 (рекомендация зафиксирована там).
- **Публичные маршруты** остаются для сценария «пользователь зашёл → /start → оплата / поддержка», иначе белый список сделает бота непроходимым для новых.

---

## 2. Термины

- **Публичный маршрут** — команда или callback, для которого middleware **не** требует лицензию (как сейчас «public» у `BanBlacklistPolicy`).
- **Закрытый маршрут** — всё остальное в личке: диалог с агентом, медиа, callback не из whitelist префиксов и т.д.
- **Действующая лицензия** — запись в таблице `license`, по правилам **`get_user_active_license(user_id)`** из `storage/db/licenses.py`: `status = 'active'` и `expires_at > NOW()` (и актуализированная логика истечения, уже заложенная в mixin, в т.ч. перевод в `expired` при чтении в `get_user_license_info` где применимо).

---

## 3. Источник истины для «есть доступ»

### 3.1. Рекомендуемый критерий (обязательно унифицировать в коде)

Один метод на уровне `UserStorage` / `Database` (название на усмотрение реализации, смысл фиксируется здесь):

```text
async def user_has_active_license(user_id: int) -> bool
```

**TRUE**, если существует действующая лицензия в смысле **того же SQL/логики**, что и `get_user_active_license` (не только `status`, но и **срок**).

### 3.2. Устаревшие проверки

- Метод `check_user_license` в `users.py` (только `status = 'active'`) **не использовать** для доступа к боту без доработки; либо удалить дублирование, либо переписать так, чтобы семантика совпадала с `get_user_active_license`.

### 3.3. Выдача лицензии

Существующие потоки **не отменяются:** оплата (`create_or_extend_license` в fulfillment), реферальные бонусы (`extend_license_by_days`), ручное добавление админом — по-прежнему создают/продлевают строку в `license`. После перехода на белый список именно эта строка (при соблюдении срока) открывает доступ.

## 4. Политика доступа (новый класс)

### 4.1. Размещение

Реализовать новую политику в `bot/access/policies.py` (например `LicenseWhitelistPolicy` или `ActiveLicensePolicy`), реализующую `AccessPolicy`.

### 4.2. Алгоритм `decide`

Вход: `AccessContext`, `event_obj` (как сейчас).

1. Если маршрут **публичный** (п. 4.4) → `AccessDecision.ALLOW_PUBLIC`.
2. Загрузить пользователя (`get_user`). Если **`is_banned`** → `DENY` (независимо от лицензии).
3. Если **`user_has_active_license(user_id)`** → `ALLOW`.
4. Иначе → `DENY`.

Опционально для отладки: логировать `event_type`, `user_id`, причину `no_license` vs `banned` (без персональных данных сообщения).

### 4.3. Забаненные и публичные маршруты

**Рекомендация для единообразия:**

- Для **публичных** команд забаненный пользователь может получать ответы только там, где нужно для юридической/продуктовой нейтральности (/start с текстом «доступ ограничен»).  
- Практичный минимум v1: **публичные маршруты для забаненных** вести себя как сейчас у `BanBlacklistPolicy` для public (т.е. бан **не** блокирует только public) **или** блокировать и public при бане — **выбрать одно поведение** и описать в release notes.

**Зафиксированное требование из текущего ТЗ:** полный доступ только по лицензии; **бан** — явный запрет использования закрытых функций. Комбинация «забанен, но с лицензией» → закрытые маршруты **DENY** (п. 4.2).

### 4.4. Список публичных маршрутов

Сохранить механизм `public_commands` и `public_callback_prefixes` как у `BanBlacklistPolicy` (передача извне в конструктор политики).

**Минимальный набор (уточнить перед релизом по факту регистрации команд в `command_handlers.py` / фичах):**

| Тип | Значения (пример, подставить фактические из проекта) |
|-----|-----------------------------------------------------|
| Команды | `/start` — обязательно; типично также `/payment`, `/donat`, `/support`, при использовании — `/subs`, `/feedback`, `/club` и др., **если** они должны быть доступны **до** покупки лицензии. |
| Callback prefix | Префиксы для оплаты и онбординга: например `payment_` (см. исключение в `MessageHandlers` для callback), и другие, которые должны работать без лицензии. |

**Правило:** любая команда, без которой пользователь **не сможет** оплатить или написать в поддержку, **должна** быть в публичном списке. Всё остальное — закрыто до появления активной лицензии.

### 4.5. Подключение в приложении

В `bot/base_app.py` заменить:

```python
BanBlacklistPolicy(self.user_storage)
```

на экземпляр новой политики с тем же `user_storage` и согласованным списком публичных команд/prefixes.

Класс `BanBlacklistPolicy` **не обязательно удалять** — можно оставить для других деплоев или тестов; для целевого продукта по умолчанию используется политика по п. 4.2.

## 5. `LicenseService` (`bot/features/license_service.py`)

Сейчас сервис назван «лицензия», но проверяет бан. Требуется привести в соответствие с ТЗ.

### 5.1. `check_access(user_id)`

- **TRUE:** есть действующая лицензия (`user_has_active_license`) **и** пользователь не забанен.
- **FALSE:** нет лицензии или бан.

### 5.2. `can_ask_question(user_id)`

- Возвращать `AccessResult` с `reason` вида `no_active_license` | `user_banned` | `access_granted` | `error`.

### 5.3. `get_access_info(user_id)`

- `has_license` — наличие **активной неистёкшей** лицензии (не инверсия бана).
- Убрать заглушки «999999 вопросов», если они не используются продуктом; либо оставить с явным комментарием «legacy / не используется».

### 5.4. Потребители

Найти все вызовы `LicenseService` / `check_access` / `can_ask_question` по репозиторию и убедиться, что после смены семантики UI и сообщения пользователю согласованы (например текст «оформите подписку» вместо «заблокированы» где уместно).

## 6. Middleware и личка / группы

### 6.1. Личка

`InboundLoggingMiddleware` добавляет в `data` ключи `access_event_type`, `access_event_obj`, `access_user_id` **только** для апдейтов, цель которых — private chat (`_inbound_target_is_private`). Для таких же апдейтов срабатывает `AccessControlMiddleware`.

**Требование:** для апдейтов **без** этих ключей (например, группы) `AccessControlMiddleware` уже передаёт управление дальше — поведение сохранить.

### 6.2. Группы и каналы

Если в проекте позже появится RAG/индексация в группе, доступ в группе регулируется отдельно (по `chat_id`), а не этим whitelist в личке. Текущее ТЗ **не** меняет правила для супергрупп, если только они явно не проходят через тот же pipeline с `access_*` в `data`.

## 7. Сообщение при отказе

### 7.1. Текст

Сейчас используется `DENY_MESSAGE_HTML_DEFAULT` про блокировку. Для отказа **по отсутствию лицензии** целесообразно:

- либо **разные** сообщения для «забанены» и «нет подписки» (предпочтительно);
- либо один нейтральный текст «доступ к функциям бота ограничен» с ссылкой на `/payment` и поддержку.

**Требование:** в конструкторе `AccessControlMiddleware` или в политике предусмотреть возможность передать **два** HTML-шаблона либо callback для выбора текста по причине отказа.

Минимум v1: одно сообщение, но текст не должен утверждать «ваш аккаунт заблокирован», если причина — отсутствие лицензии.

## 8. Критерии приёмки

1. Пользователь **без** строки в `license` или с истёкшим `expires_at` **не** может отправить сообщение в личку, которое попадает в закрытые хендлеры (после прохождения media/route) — получает отказ и **не** вызывает диалог с агентом.
2. Пользователь с **действующей** лицензией (`get_user_active_license` not None) и **без** бана — полный доступ к закрытым маршрутам.
3. Пользователь **забанен** — закрытые маршруты недоступны всегда.
4. Публичные команды из согласованного списка работают **без** лицензии.
5. `LicenseService` описывает именно лицензию + бан, а не «все кроме бана».
6. Регистрация фич в `bot_app.py` может быть сокращена **без удаления файлов** фич; сборки с полным и урезанным набором фич остаются возможными.

## 9. Миграция и эксплуатация

### 9.1. Существующие пользователи

После включения белого списка все пользователи **без** активной неистёкшей лицензии потеряют доступ к закрытым функциям.

**Операционные варианты (вне кода или одноразовый скрипт):**

- массово выдать лицензии (например, на год) тем, кто считался активным до миграции;
- или: перед включением убедиться, что все нужные `user_id` уже имеют строки в `license` с корректным `expires_at`.

### 9.2. Мониторинг

- Логировать отказы по `no_license` с усечённым контекстом для оценки конверсии и ошибок конфигурации публичных команд.

## 10. Не входит в объём данного ТЗ

- Удаление или переписывание фич `referral`, `support` и т.д.
- Изменение схемы таблицы `license` (если потребуется отдельная таблица «приглашения» — отдельное ТЗ).
- RAG / Chroma / групповая индексация (см. общий `tz.md` проекта) — пересекается только тем, что доступ в личке определяется этим документом.
