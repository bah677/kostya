# Единое ТЗ: модернизация текущей сборки `avatar`

**Назначение:** один документ описывает, как развивать **именно эту** кодовую базу (не «бот с нуля»): что уже есть, что добавить по продукту (RAG / банк идей), как изменить контроль доступа, как устроены слои.  
**Исходные требования по продукту:** см. также корневой [`tz.md`](../tz.md) (детали RAG, чанкинг, группа с топиками, `/generate`, золотой фонд). Здесь — привязка к репозиторию и приоритеты внедрения.

**База данных:** отдельного проекта «миграция с продакшена» **нет**. Таблицы и схема PostgreSQL **создаются и расширяются по мере разработки** (SQL/Alembic в репозитории по мере появления фич). Ниже где уместно — логические сущности, а не обязательный перечень уже существующих таблиц.

## 2. Целевая модернизация (два крупных направления)

### Направление A — Продукт: RAG и контент-ассистент

Содержательно совпадает с [`tz.md`](../tz.md):

- **ChromaDB** (PersistentClient), каталог данных из конфига (например `./chroma_data`).
- Коллекции **`expert_materials`**, **`golden_examples`**; эмбеддинги OpenAI `text-embedding-3-small` (1536).
- **Индексация** из одной разрешённой **супергруппы** с **топиками**: фильтр `chat_id`, извлечение текста (медиа + длинный текст ≥ 300 символов), маппинг топика → `product` / `type`, теги через LLM, чанкинг (500–800 токенов, overlap 100), дедуп, **reply** с отчётом.
- **`retrieve_context(query, top_k, filter)`** — текст для промпта.
- **Режим генерации** в личке: `/generate &lt;тема&gt;` — retrieval + golden (top_k≈2), отдельный system prompt, **без** истории чата; обычный режим — как сейчас через `AgentsClient.run`.
- **👍 / 👎** под ответами режима генерации → запись в `golden_examples`.

**Новые модули** (логические имена; фактический пакет — по соглашению в репо, например `bot/rag/` или корневой `knowledge/`):

- инициализация Chroma и коллекций;
- embedding / retriever / golden_store;
- `group_indexer` (пайплайн группы + вызов `MediaProcessor`).

**Конфиг:** минимум `GROUP_CHAT_ID`, путь Chroma, ключи OpenAI для эмбеддингов; опционально фильтр топиков, маппинг названий топиков.

### Направление B — Доступ: белый список по лицензии

Детально зафиксировано в [`TZ_LICENSE_WHITELIST_ACCESS.md`](TZ_LICENSE_WHITELIST_ACCESS.md). Кратко:

- Закрытые маршруты в личке — только при **действующей лицензии** (критерий как `get_user_active_license`: активный статус и не истёк срок) и при отсутствии **бана**.
- Публичные команды/callback (как минимум `/start`, оплата, поддержка — список согласовать с `AppCommandHandlers` и callback-префиксами, напр. `payment_`) остаются доступны **без** лицензии.
- Реализация: новая политика в `bot/access/policies.py`, замена в `AccessControlMiddleware`; перепись **`LicenseService`** под реальную семантику лицензии; при необходимости разные тексты отказа (бан vs нет подписки).

## 3. Согласование публичных маршрутов и RAG

После ввода белого списка любая команда, необходимая **до** выдачи лицензии (в т.ч. оплата и `/start`), должна быть в списке публичных у политики доступа.

Режим **`/generate`** — **закрытый** (требует лицензию), как и обычный диалог с агентом.

Групповая индексация — по **`GROUP_CHAT_ID`** и политике бота в группе (участники — авторы материалов); доступ к **личке** по лицензии индексатор не заменяет.

## 4. Логирование и наблюдаемость

Существующие: [`bot/middleware/inbound_logging.py`](../bot/middleware/inbound_logging.py), [`bot/logging/`](../bot/logging/), исходящий лог.

Дополнить событиями: индексация (принят/пропуск события, чанки, дубликат), retrieval, вызов режима generate, ошибки Chroma/LLM. Корреляция по `user_id` / `chat_id` / `message_thread_id` где применимо.

## 5. Порядок работ (рекомендуемый)

Порядок можно смещать по приоритету продукта; логичная последовательность:

1. **Доступ по лицензии** (Направление B) — иначе тестировать RAG в личке на всех пользователях нежелательно.
2. **Chroma + коллекции + embedding + retriever** (без группы).
3. **Групповой индексер** + интеграция `MediaProcessor` + отчёт в треде.
4. **Ветка `/generate` в `AgentsClient` / `MessagingFeature`** + промпты + `retrieve_context` + golden store.
5. **Inline 👍/👎** и запись в `golden_examples`.
6. Полировка: кэш названий топиков, дедуп, тексты ошибок.

Таблицы PostgreSQL добавляются **по мере необходимости** (если решите хранить дедуп, очереди, служебные связи — в том же стиле mixin'ов, без отдельного «большого» миграционного релиза).

## 6. Критерии готовности (сводные)

| Область | Критерий |
|---------|-----------|
| RAG | Как в разделе 7 [`tz.md`](../tz.md): индексация в группе с отчётом, `/generate` с контекстом и golden, обычный чат не сломан, 👍/👎 влияют на few-shot |
| Доступ | Закрытые функции только с действующей лицензией и без бана; публичные маршруты согласованы; сообщения отказа не путают бан и отсутствие подписки (если реализованы два текста) |
| Сборка | Фичи не удалены; опциональная регистрация в `BotApplication` сохранена |
| БД | Схема вводится итеративно; требований к миграции из внешней/старой БД нет |
