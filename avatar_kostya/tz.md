# Техническое задание: Доработка Telegram-бота – «Банк идей и контент-ассистент» (RAG-система)

## 1. Контекст и цель

У нас есть работающий Telegram-бот, который:
- Хранит историю диалогов в PostgreSQL, логирует взаимодействия.
- Общается с LLM (OpenAI / DeepSeek) через асинхронные методы.
- Имеет `Медиапроцессор`, извлекающий текст из любых медиаматериалов (голосовые, файлы, скриншоты).
- Предоставляет метод `run(user_message, user_id)` для получения ответа от LLM.

**Цель доработки:** превратить бота в «цифрового двойника» эксперта (психолога-расстановщика), который генерирует продающие сторис/посты, опираясь на загруженные материалы (вебинары, курсы, скриншоты успешных сторис, диалоги). Ключевая идея: **наполнение базы знаний происходит через специальную Telegram-группу с топиками**, куда автор и ассистент скидывают материалы в фоне. Бот автоматически индексирует их в RAG-хранилище и присылает короткий отчёт.

## 2. Текущая архитектура (для понимания)

```
Пользователь (личка) → бот → run()
    ├─ получаем историю диалога из PostgreSQL
    ├─ формируем messages = [system_prompt] + history + [user_message]
    ├─ вызываем LLM (OpenAI / DeepSeek)
    └─ возвращаем ответ, логируем
```

Медиапроцессор: на вход подаётся файл → на выходе извлечённый текст.

Новое: параллельно работает слушатель сообщений в группе.

## 3. Что надо добавить

### 3.1. Векторная база знаний (ChromaDB)

- Установить ChromaDB в режиме встроенной персистентной БД (`PersistentClient`).
- Коллекция `expert_materials` с использованием OpenAI `text-embedding-3-small` (размерность 1536). Параметр `embedding_function=OpenAIEmbeddingFunction(...)` позволяет автоматически вычислять эмбеддинги при добавлении.
- Метаданные для каждого чанка:
  - `source` (имя файла или дата текста)
  - `type` (`webinar_transcript`, `course_text`, `story_screenshot`, `dialog`, `manual_text`)
  - `product` (из названия топика: `individual_consultation`, `group_session`, `course_basics` и т.д.)
  - `tags` (ключевые темы, извлечённые автоматически, например, `страх перемен`, `возражение "дорого"`)
  - `added_by` (user_id загрузившего, опционально)
- Размер чанка: 500-800 токенов, перекрытие 100 токенов.

### 3.2. Автоматическая индексация из Telegram-группы (основной способ наполнения)

**Предпосылка:** владелец бизнеса создаёт отдельную приватную группу. В ней он создаёт **топики** (тем самым структурируя материалы):
- «Продающие сториз курс»
- «Продающие сториз консультации»
- «Курс “Основы расстановок”»
- «Курс “Продвинутые техники”»
- «Диалоги / возражения»
- и т.д.

В эту группу добавляется бот, автор и ассистент. Права бота: чтение сообщений и отправка сообщений в топики.

**Логика работы бота в группе:**

1. **Фильтр сообщений.** Бот реагирует только на сообщения из разрешённой группы (проверка `chat_id` из конфигурации). Сообщения из лички обрабатываются отдельно (см. 3.4).
2. **Извлечение контента.** Если сообщение содержит:
   - **Файлы:** документ, аудио, видео, фото – обрабатываются через медиапроцессор для получения текста.
   - **Текст:** если длина текста ≥ 300 символов, он тоже индексируется как самостоятельный материал (например, автор вручную написал пост в топик).
3. **Определение метаданных по топику:**
   - Бот получает `message_thread_id` (ID топика).
   - Кэширует названия топиков (через `get_chat()`) для быстрого доступа, чтобы не дёргать API на каждое сообщение.
   - Название топика становится основой для полей `product` и `type`. Пример маппинга:  
     - Топик «Продающие сториз консультации» → `type = story`, `product = consultation`  
     - Топик «Курс “Основы расстановок”» → `type = course_material`, `product = course_basics`  
     - Топик «Диалоги / возражения» → `type = dialog`, `product = general`  
   - Можно предусмотреть префиксы в названии топика для уточнения типа (например, `[sales] Продающие сториз курс`).
4. **Определение тегов через LLM:**
   - После получения текста бот вызывает быструю/дешёвую LLM (например, DeepSeek или GPT-4o-mini) с промптом:  
     > «Выдели 3–5 ключевых тем из этого текста. Перечисли их через запятую. Темы должны быть краткими: “страх перемен”, “работа с возражениями”, “кейс Марины”»
   - Результат сохраняется в `tags`.
5. **Чанкинг и индексация:**
   - Текст разбивается на чанки с учётом метаданных.
   - Каждый чанк добавляется в коллекцию `expert_materials` через `collection.add(...)`.
   - Перед добавлением проверяется, нет ли дубликата: по хешу контента или по комбинации `source + номер чанка`. Если есть – пропускаем и сообщаем «уже есть».
6. **Ответ-отчёт в топик:**
   - Бот отправляет короткое сообщение-ответ на проиндексированное сообщение (reply):  
     > ✅ Загружено: `[имя файла или первые 50 символов текста]`  
     > Чанков: 24  
     > Темы: страх перемен, ценность, работа с возражениями  
     > Продукт: Консультации  
     > Добавил: @username  
   - Если материал уже был в базе:  
     > ℹ️ Этот материал уже есть в базе знаний (добавлен ранее).

**Преимущества:**
- Не требует от эксперта выполнения команд, всё работает в фоне.
- Естественная среда общения (Telegram), привычная для владельца и ассистента.
- Автоматическая структуризация по топикам без ручной разметки.

### 3.3. Retrieval (поиск по смыслу)

Метод `retrieve_context(query: str, top_k: int = 5, filter: Optional[dict] = None) -> str`:
- Вызов `collection.query(query_texts=[query], n_results=top_k, where=filter)`.
- Формирование текстового блока: `[source - tags]: текст_чанка`
- Возврат контекста для вставки в промпт.

### 3.4. Маршрутизация в личном чате с ботом (run)

Модифицировать `run()` (или добавить обработчик команд):
- **Обычный чат** – работает как прежде, с историей диалога.
- **Генерация контента** – активируется командой `/generate <тема>` или ключевыми словами.
  - Извлекается тема → вызов `retrieve_context(topic)`.
  - Параллельный поиск в коллекции `golden_examples` (см. 3.5).
  - Полученный контекст подставляется в системный промпт для контента (плейсхолдер `{retrieved_context}`).
  - Используется отдельный системный промпт (экспертный стиль, требование опираться на контекст).
  - История диалога не используется.
  - Вызов LLM → возврат ответа.

### 3.5. Золотой фонд (few-shot примеры)

- Вторая коллекция в ChromaDB: `golden_examples`.
- Под каждым ответом в режиме генерации – inline-кнопки: 👍 / 👎.
  - 👍 → сохраняем пару «тема → ответ» в коллекцию с эмбеддингом темы.
  - 👎 → опционально запрашиваем причину, не сохраняем.
- При следующей генерации делаем запрос в `golden_examples`, top_k=2, подставляем в промпт как few-shot образцы.

### 3.6. Интеграция и структура кода

- Новые компоненты оформить как отдельные сервисы:
  - `vector_db.py` – инициализация ChromaDB, коллекции.
  - `embedding_service.py` – обёртка для OpenAI эмбеддингов.
  - `retriever.py` – поиск по основной коллекции и золотому фонду.
  - `group_indexer.py` – слушатель сообщений группы, логика индексации.
  - `golden_store.py` – операции с золотым фондом.
- Старый `run()` получает ветку для `/generate`.
- В конфиг добавить `GROUP_CHAT_ID` (разрешённая группа) и, возможно, список ID топиков, если нужна дополнительная фильтрация.

## 4. Технические детали и ограничения

- **Эмбеддинги:** OpenAI `text-embedding-3-small`, размерность 1536.
- **ChromaDB:** версия ≥0.4, персистентная папка `./chroma_data`.  
- **LLM:** генерация остаётся на DeepSeek/OpenAI. Для автоизвлечения тегов можно использовать ту же или отдельную облегчённую модель.
- **Telegram API:** обработка `message_thread_id`, кэширование названий топиков через `bot.get_chat()`. Библиотека aiogram или python-telegram-bot.
- **Дубликаты:** проверка по хешу MD5 первых 200 символов + имя файла.
- **Безопасность:** бот обрабатывает сообщения только из группы с `chat_id` из env. В личке – обычное взаимодействие.

## 5. Клиентский путь (как это выглядит для эксперта)

**Первоначальная настройка:**
1. Эксперт создаёт группу «Материалы для бота», добавляет ассистента и бота.
2. Создаёт топики под каждый продукт/тип контента.
3. Бот однократно получает ID группы (через команду `/register_group` администратором или через конфиг).

**Повседневное использование:**
- Автор видит успешную сторис → кидает скриншот в топик «Продающие сториз консультации».
- Ассистент загружает аудио вебинара в «Курс “Основы”».
- Бот в ответ пишет краткий отчёт (чанков, темы).
- Эксперт в личке с ботом пишет `/generate прогрев на тему страха перемен` → получает серию сторис.
- Лайкает удачные варианты, они попадают в золотой фонд.

## 6. План реализации

1. Установка и настройка ChromaDB, создание коллекций (1 час).
2. Сервисы эмбеддингов, retrieval (2 часа).
3. Разработка `group_indexer`: слушатель сообщений, обработка файлов и текста, интеграция с медиапроцессором, извлечение топиков и тегов, проверка дубликатов, отправка отчёта (4-5 часов).
4. Модификация `run()` для маршрутизации команд и генерации контента (2-3 часа).
5. Реализация кнопок 👍/👎 и золотого фонда (2 часа).
6. Тестирование сценариев в группе и личке (2-3 часа).
Общая длительность: около 2 недель (в рамках заявленного бюджета).

## 7. Критерии готовности

- Сообщения с файлами или длинным текстом в топиках указанной группы автоматически индексируются; бот отвечает в тред отчётом.
- По команде `/generate <тема>` бот генерирует контент на основе индексированных материалов.
- Кнопки 👍/👎 работают; одобренные ответы сохраняются и влияют на будущие генерации.
- Обычный чат-режим не нарушен.
